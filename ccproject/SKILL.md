# Project Awareness: Three-Layer Context Architecture

You cannot hold an entire non-trivial codebase in one context window. Stop trying.
Instead, maintain a structured knowledge system that gives you the 30,000-foot view
always, subsystem details on demand, and a compressed structural map for navigation.

## The Three Layers

### Layer 1: Constitution (CLAUDE.md — always loaded, ~150-200 lines)
The project's CLAUDE.md is the "hot memory." It loads on every session start. It contains:
- What the project IS (2-3 sentences)
- Module/subsystem inventory (name + one-line purpose each)
- Key architectural invariants (the rules that must NEVER be violated)
- Design tensions (known conflicts in the architecture, e.g., "block cache is 4K but
  AG headers are 512-byte sectors — every write to block 0 must use sector granularity")
- Routing table: which subsystem doc to load for which task
- Build/test commands
- The [AWARENESS] marker with metadata (last update, coverage stats)

This document is the SINGLE MOST IMPORTANT artifact. If an agent does nothing else,
reading this document should prevent the most common mistakes.

### Layer 2: Subsystem Docs (.claude/awareness/subsystems/*.md — loaded on demand)
Each major subsystem gets its own document. These are the "domain expert" docs. They contain:
- What the subsystem does and its public API
- Internal architecture (key data structures, state machines, flows)
- Invariants specific to this subsystem
- Known pitfalls and historical bugs (things that were tried and FAILED)
- Cross-subsystem interactions (what this subsystem assumes about others)
- File list (which source files belong to this subsystem)

These load when Claude enters that subsystem's code. Subdirectory CLAUDE.md files can
reference these, or the routing table in the root CLAUDE.md directs Claude to read them.

### Layer 3: Structural Map (.claude/awareness/structural-map.md — loaded on demand)
A compressed representation of the entire codebase's structure. Generated by parsing
source files and extracting:
- All function/method signatures (name, parameters, return type)
- All struct/class/type definitions (name, key fields)
- Call graph edges (who calls whom)
- File-to-symbol mapping (which symbols live in which files)

This is NOT the full source code. It's the skeleton — enough for an agent to understand
how pieces connect without reading every file. Target: <5% of the raw codebase token count.

Format uses a compact notation:
```
[file.c]
  fn mxfs_cache_write(cache*, block_id, buf*, len) -> int
    calls: bdev_write, lock_acquire, lock_release
    called_by: alloc_init, format_ag, mount_replay
  struct mxfs_cache_entry { block_id, refcount, dirty, data[BLOCK_SIZE] }
```

## Bootstrap Protocol (First Time Setup)

When a project has no awareness infrastructure, or the user asks to bootstrap:

### Phase 1: Reconnaissance

Run the project analyzer to get a comprehensive overview:

```bash
python3 ~/.claude/skills/project-awareness/scripts/analyze_project.py ./ --verbose
```

This auto-detects languages, build systems, subsystem boundaries, and inter-subsystem
dependencies. Review the JSON output to understand the project structure.

Also read existing CLAUDE.md, README, and top-level documentation for context the
analyzer can't capture (design intent, invariants, history).

### Phase 2: Subsystem Identification

The analyzer output includes a `subsystems` array with auto-detected subsystems.
Review these and adjust:
- Merge subsystems that are tightly coupled (aim for 3-7 total)
- Split subsystems that serve multiple purposes
- **Present the subsystem list to the user for confirmation before proceeding.**

For each subsystem, the analyzer provides:
- Its root directory/files
- Its public API (exported functions/types)
- Its dependencies on other subsystems

AIM FOR 3-7 SUBSYSTEMS. If you identify more than 7, some should be merged.
If fewer than 3, the project may not need subsystem-level docs (just use a detailed CLAUDE.md).

### Phase 3: Generate the Structural Map

Run the structural map generator:

```bash
python3 ~/.claude/skills/project-awareness/scripts/generate_structural_map.py ./ --verbose
```

This automatically:
- Parses all source files (C, C++, Python, Go, Rust, JS/TS)
- Extracts function signatures, struct/type definitions, call graph edges
- Uses tree-sitter for accurate AST parsing when available, falls back to regex
- Writes `.claude/awareness/structural-map.md` in compact notation format
- Reports approximate token count

If repomix is available (`npx repomix --version`), you can also use `repomix --compress`
for additional structural compression (~70% token reduction).

Mermaid call graph diagrams are available on request via `--mermaid` flag or the
`generate_mermaid_callgraph.py` script, but are not generated by default.

### Phase 4: Generate Subsystem Docs

For each identified subsystem, read its key files (headers first, then implementation)
and generate a doc following the subsystem template at
`~/.claude/skills/project-awareness/templates/subsystem.md.template`.

DO NOT try to be exhaustive. Capture:
- What a developer (or agent) needs to know to work in this subsystem safely
- What mistakes are easy to make (and have been made before)
- How this subsystem interacts with others

### Phase 5: Generate the Constitution (CLAUDE.md)

Write or update the project's root CLAUDE.md following the CLAUDE.md template.
This must stay under 200 lines. If the project already has a CLAUDE.md, PRESERVE
existing content and ADD the [AWARENESS] section.

### Phase 6: Generate Slash Commands

Copy from `~/.claude/skills/project-awareness/templates/commands/` to `.claude/commands/project/`:
- `bootstrap-awareness.md`
- `refresh-map.md`
- `update-awareness.md`

### Phase 7: Report

Tell the user:
- What subsystems were documented
- How many tokens the structural map uses
- Any gaps or areas that need human input (invariants you couldn't determine
  from code alone — these ALWAYS exist, ask the user)
- Suggest the user review the invariants section especially

## Session Protocol (Every Session)

When starting a session on a project with awareness infrastructure:

1. CLAUDE.md loads automatically (Layer 1 — always happens).
2. Read the [AWARENESS] section. Note the last-updated date.
3. If the structural map is stale (>7 days or the user says things changed),
   suggest running `/project:refresh-map`.
4. When the user describes a task, use the ROUTING TABLE to identify which
   subsystem(s) are involved.
5. Read ONLY the relevant subsystem doc(s) (Layer 2).
6. If you need to understand cross-cutting dependencies, read the structural
   map (Layer 3).
7. NOW begin work, following deep-focus protocol if the task is complex.

## Update Protocol (While Working)

THIS IS CRITICAL. The awareness docs are only useful if they stay current.

### Automated enforcement (hooks)

When ccproject is installed, three hooks keep the docs honest so this isn't
purely a matter of remembering:

- **PostToolUse** records every source file and awareness doc you touch this
  session.
- **Stop** auto-regenerates Layer 3 (the structural map — pure extraction, no
  judgment needed) and then **blocks the session from ending** if you changed
  code in a subsystem but did not update that subsystem's doc. The block names
  the exact docs to fix. It backs off after `CCPROJECT_MAX_NUDGES` re-feeds
  (default 3) so a genuinely-stuck session can still exit; `CCPROJECT_NO_ENFORCE=1`
  disables blocking, `CCPROJECT_NO_AUTOREGEN=1` disables the map regen.
- **SessionStart** reports which subsystems have source newer than their doc.

The hooks handle Layer 3 entirely and *force* you to keep Layers 1–2 current.
They cannot write the prose for you — that judgment is still yours. The steps
below are what you do when the Stop hook blocks (or, better, before it has to).

### During a session:
- If you discover a NEW invariant or design constraint -> note it for the CLAUDE.md
- If you discover a pitfall or historical bug -> note it for the subsystem doc
- If you add/remove/rename a public function -> note it for the structural map
- If you create a new file -> note which subsystem it belongs to

### At session end (or before context handoff):
1. Review your notes from above.
2. Update the relevant subsystem doc(s) with new information.
3. If invariants or the routing table changed, update CLAUDE.md.
4. Do NOT regenerate the structural map for minor changes — just note what changed.
   Regenerate only on major refactors or when the user asks.
5. Update the [AWARENESS] last-modified date.

### For Agent Teams:
- The TEAM LEAD loads Layer 1 (CLAUDE.md) + Layer 3 (structural map).
- Each WORKER loads Layer 1 (CLAUDE.md) + their assigned Layer 2 (subsystem doc).
- When a worker makes changes, it MUST update its subsystem doc before reporting
  completion to the team lead.
- The team lead updates CLAUDE.md if the workers' changes affect invariants or
  cross-subsystem interactions.
- Workers do NOT update CLAUDE.md directly — they report to the team lead,
  who decides what rises to the constitution level.

## Quality Standards for Awareness Docs

### CLAUDE.md (Constitution)
- MUST be under 200 lines
- MUST have the [AWARENESS] section with metadata
- MUST have the routing table
- Every invariant listed must explain WHY it exists, not just WHAT it is
- Test: A fresh Claude Code session reading ONLY this file should be able to
  answer "what are the 3 most dangerous things I could accidentally break?"

### Subsystem Docs
- SHOULD be 50-150 lines each
- MUST list all public API functions with one-line descriptions
- MUST list known pitfalls
- MUST list cross-subsystem dependencies
- Test: An agent assigned to this subsystem should be able to work safely
  after reading ONLY the CLAUDE.md + this doc

### Structural Map
- SHOULD be <5% of total source code token count
- MUST include all public function signatures
- MUST include all type/struct definitions
- SHOULD include call graph edges for key functions
- Test: Given a function name, you should be able to find who calls it
  and what it calls without reading source files

## When Awareness is Insufficient

If you're working and realize the docs don't cover something important:

1. DON'T just ignore it and read the source directly.
2. DO read the source to understand it.
3. THEN update the appropriate awareness doc with what you learned.
4. This is how the system gets better over time — every session that touches
   a gap should fill it.
